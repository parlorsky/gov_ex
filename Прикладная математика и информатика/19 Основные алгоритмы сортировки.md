# Основные алгоритмы сортировки. Их сложность, достоинства и недостатки. Алгоритм быстрого поиска в отсортированном массиве. Сложность поиска в отсортированном и не отсортированном массиве.

## TL;DR
**Простые сортировки** — пузырьковая, вставками, выбором — $O(n^2)$, in-place, понятны. **Эффективные сравнительные** — quicksort (средне $O(n\log n)$, худшее $O(n^2)$), mergesort ($O(n\log n)$ всегда, $O(n)$ память, стабильна), heapsort ($O(n\log n)$, in-place). **Timsort** — гибрид merge+insertion, используется в `sorted/list.sort` Python и Java. **Линейные сортировки** (counting, radix, bucket) — $O(n)$, но требуют ограничений на ключи. Нижняя граница для сравнительной сортировки — $\Omega(n\log n)$. Поиск в **неотсортированном** массиве — $O(n)$ (линейный); в **отсортированном** — $O(\log n)$ **бинарный поиск**.

## Развёрнуто

### Сравнительные критерии
- **Время** в среднем / худшем / лучшем.
- **Память**: in-place ($O(1)$ или $O(\log n)$ на стек) или $O(n)$.
- **Стабильность**: сохраняется относительный порядок равных элементов.

### Простые сортировки ($O(n^2)$)

**Пузырьковая (bubble sort)**. Многократно проходим массив, обмениваем соседей в неверном порядке. $O(n^2)$, in-place, стабильна. Лучшее $O(n)$ при флаге «не было обменов».

**Сортировка вставками (insertion sort)**. Каждый элемент вставляется в правильное место уже отсортированного префикса. $O(n^2)$ в среднем, $O(n)$ на почти отсортированных, in-place, стабильна. Хороша для малых $n$ или дополнения к более сложным алгоритмам.

**Сортировка выбором (selection sort)**. Многократно ищем минимум в неотсортированной части и переносим в начало. $O(n^2)$ всегда, in-place, **не стабильна** в стандартной реализации.

### Эффективные сортировки

**Quicksort** (Хоар, 1960). Выбирается опорный элемент (pivot), массив делится на меньшие и большие; рекурсия. Время: средне $O(n\log n)$, худшее $O(n^2)$ (плохой выбор pivot — отсортированный массив). In-place ($O(\log n)$ на стек), **не стабильна**. На практике быстрейший алгоритм. Защита от худшего случая: рандомизированный выбор pivot, или **introsort** (переключение на heapsort при глубокой рекурсии).

```python
def quicksort(a, lo=0, hi=None):
    if hi is None: hi = len(a) - 1
    if lo < hi:
        p = partition(a, lo, hi)
        quicksort(a, lo, p-1)
        quicksort(a, p+1, hi)
```

**Mergesort** (фон Нейман). Делим массив пополам, рекурсивно сортируем, сливаем. $O(n\log n)$ всегда, $O(n)$ дополнительной памяти, **стабильна**. Хорошо работает на связных списках и внешней памяти.

```python
def mergesort(a):
    if len(a) <= 1: return a
    m = len(a) // 2
    L, R = mergesort(a[:m]), mergesort(a[m:])
    return merge(L, R)
```

**Heapsort**. Строим max-heap из массива ($O(n)$), затем $n$ раз извлекаем максимум. $O(n\log n)$ всегда, in-place, **не стабильна**. Стабильно медленнее quicksort на практике, но даёт гарантированную верхнюю границу.

**Timsort**. Гибрид merge + insertion. Ищет уже отсортированные «прогоны» (runs) и сливает их. $O(n\log n)$ худшее, $O(n)$ на почти отсортированных, **стабильна**. Используется в `sorted` Python и `Arrays.sort` Java для объектов.

### Линейные сортировки (с ограничениями на ключи)

**Counting sort**. Если ключи — целые в $[0, k)$, считаем частоты и восстанавливаем массив. $O(n+k)$. Хороша при малых $k$, стабильна.

**Radix sort**. Разрядная сортировка целых/строк фиксированной длины. $O(d\cdot(n+b))$, $d$ — число разрядов, $b$ — основание. Использует counting sort как стабильную внутреннюю.

**Bucket sort**. Распределяем элементы по «корзинам» по диапазонам и сортируем корзины. $O(n+k)$ в среднем для равномерных данных.

### Нижняя граница
Для **сравнительных** сортировок дерево решений имеет $n!$ листьев, высота $\ge \log_2 n! = \Omega(n\log n)$. Линейные сортировки обходят это, используя информацию о структуре ключей.

### Поиск в массиве
**Линейный поиск** в неотсортированном массиве: проходим до первого совпадения. $O(n)$ в худшем и среднем случае. Не требует подготовки данных.

**Бинарный поиск** в отсортированном массиве: на каждом шаге сравниваем со средним элементом и идём в нужную половину.
```python
def bsearch(a, x):
    lo, hi = 0, len(a) - 1
    while lo <= hi:
        m = (lo + hi) // 2
        if a[m] == x: return m
        elif a[m] < x: lo = m + 1
        else: hi = m - 1
    return -1
```
Сложность: **$O(\log n)$**, $O(1)$ память. Требует упорядоченности и быстрый случайный доступ (массив, не список).

**Сравнение**: если поиск однократный — линейный; если многократный — выгоднее однажды отсортировать ($O(n\log n)$) и потом каждый запрос за $O(\log n)$. Альтернативы: хеш-таблица $O(1)$ в среднем (без сохранения порядка), сбалансированное дерево $O(\log n)$.

### Подводные камни
- Quicksort на отсортированном массиве с pivot=первый элемент — $O(n^2)$. Решение: рандомизация / median-of-three.
- Mergesort требует $O(n)$ дополнительной памяти — не подходит для критичных по памяти задач.
- Counting/radix — линейные только при ограничении ключей.
- Бинарный поиск требует **отсортированного** массива; сортировка перед одним поиском не окупается.
