## Развёрнуто

### Триада функциональной обработки
- **Map** $f: A \to B$: применяет функцию к каждому элементу. $\mathrm{map}(f, [x_1, \dots]) = [f(x_1), \dots]$. Полностью независимая обработка — идеально параллелится.
- **Filter** $p: A \to {T, F}$: оставляет элементы по предикату. Тоже тривиально параллелится.
- **Reduce / fold**: сворачивает коллекцию ассоциативным оператором $\oplus$ к одному значению.

#### Ассоциативность — ключ к параллельному reduce
$(a \oplus b) \oplus c = a \oplus (b \oplus c)$ позволяет вычислять через **сбалансированное дерево**: глубина $O(\log n)$ вместо $O(n)$.

Примеры ассоциативных операций: $+, \times, \min, \max$, конкатенация. Не ассоциативные: $-, /$.

#### Параллельная реализация (две фазы)

1. **Локальная фаза:** на каждом partition выполняем reduce внутри → один частичный результат на partition.
2. **Глобальная фаза:** собираем частичные результаты, сворачиваем финальным reduce.

Преимущество: сетевой трафик минимальный — передаются только частичные результаты, не сырые данные.

#### Чистая функция

Функция чистая, если: возвращает результат только на основе входов + не имеет побочных эффектов (не меняет глобальное состояние, не пишет в файл).

**Зачем для параллелизма:** без чистоты — race condition при одновременном изменении общего состояния разными потоками.

#### Оптимизации композиции

- **Map fusion:** $\mathrm{map}(g) \circ \mathrm{map}(f) = \mathrm{map}(g \circ f)$ — один проход вместо двух.
- **Filter fusion:** $\mathrm{filter}(p) \circ \mathrm{filter}(q) = \mathrm{filter}(p \land q)$.

#### Dask.Bag

Ленивая параллельная коллекция произвольных Python-объектов. Аналог PySpark RDD для Python.

**Создание:**

```python
b = db.from_sequence([1,2,3,4,5], npartitions=2)
b = db.read_text("logs/*.txt")
```

**Основные методы:**

- `b.map(f)` / `b.filter(p)` / `b.flatten()` / `b.distinct()`.
- `b.fold(binop, initial)` — reduce, нужна ассоциативность.
- `b.groupby(key)` — полный shuffle, дорого.
- `b.foldby(key, binop, initial)` — groupby + reduce с **локальной агрегацией** на partition'ах перед shuffle. Сильно эффективнее groupby + map.
- `b.frequencies()`, `b.topk(n, key)` — частые операции.
- `b.compute()` — запустить вычисление.

**Ленивые вычисления:** операции строят task-граф, выполняются только при `.compute()`. Позволяют: оптимизацию (fusion), экономию памяти, распределение по узлам.

#### Когда что брать

|Данные|Инструмент|
|---|---|
|Произвольные Python-объекты (JSON, dict, логи)|**Dask.Bag**|
|Таблицы (CSV, Parquet, как Pandas)|**Dask.DataFrame**|
|Числовые массивы (как NumPy)|**Dask.Array**|

#### Подводные камни

- `groupby` в Bag — full shuffle, использовать `foldby`.
- Сериализация (pickle) — узкое место, передавать простые типы.
- Map с побочными эффектами ломает корректность — только чистые функции.
- На малых данных Dask **медленнее** обычного `for` из-за overhead планировщика.

#### Главное на экзамене

1. **Map/Filter/Reduce** — определения, почему параллелятся.
2. **Ассоциативность** $\oplus$ — главное условие параллельного reduce.
3. **Две фазы reduce** — локальная + глобальная.
4. **Чистая функция** — зачем нужна для параллелизма.
5. **Dask.Bag** — ленивая коллекция с partition'ами; для неструктурированных Python-объектов.
6. **foldby vs groupby** — foldby делает локальную агрегацию на partition'ах.
